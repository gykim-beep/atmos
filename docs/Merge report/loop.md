# Atmos Mixer Pro - 개발 검증 루프 및 감독관 가이드 (`loop.md`)

본 `loop.md` 문서는 `@Maker` 에이전트가 코드를 작성하고 수정하는 과정에서 **절대적으로 준수해야 하는 자동화된 감독관(Supervisor) 문서**입니다. 코드가 단순히 오류 없이 빌드되는 것을 넘어, 앞서 정의된 7대 핵심 설계 문서(7 Pillars)와 무장애 원칙을 1픽셀, 1밀리초의 오차도 없이 지켰는지 시스템적으로 평가합니다. 

이 문서의 기준을 모두 통과하지 못한 코드는 절대로 머지(Merge)될 수 없습니다.

---

## 1. 7대 설계 문서(7 Pillars) 정합성 정밀 체크리스트

모든 작업 완료 보고 시, 아래의 9가지 대분류 질문에 대해 명확한 답변과 **객관적인 로그/코드 스크린샷 증거**를 제출해야 합니다.

1. **[PRD] 요구사항 나노 체크**:
    *   오프라인 100% 원칙을 지켰는가? (외부 API 호출 코드가 일절 없는가?)
    *   배타적 룸 독점 게이팅(Exclusive Gating) 로직이 구현되었는가? (Active가 아닌 방의 센서 패킷을 버리는 분기문이 존재하는가?)
    *   스마트 더킹(Smart Ducking) 시 150ms 하강, 300ms 상승 수식이 정확히 코드에 반영되었는가?
2. **[User Flow] 흐름 매칭 나노 체크**:
    *   앱 부팅 시 `config.json` 로딩 -> 하드웨어 장악 -> 포트 바인딩의 순서가 동기/비동기적으로 꼬임 없이 흐르는가?
    *   클리어 신호 수신 시, 즉각 현재 방을 Cleared로 바꾸고 다음 방을 Active로 자동 전환하는 상태 머신(State Machine)이 존재하는가?
3. **[Screens] 픽셀 퍼펙트 나노 체크**:
    *   전역 배경색이 정확히 `#0E0E1C` 이고 모듈 배경색이 `#161628` 로 지정되었는가?
    *   룸 클리어 버튼의 Padding 값이 정확히 가로 `16.0`, 세로 `12.0`으로 설정되었는가?
    *   Right Overflowed(RenderFlex) 에러 방지를 위해 헤더 Row 안에 `Wrap` 또는 `Expanded + SingleChildScrollView` 방어 코드가 적용되었는가?
4. **[DB Design] 스키마 충돌 방어 나노 체크**:
    *   `config.json`의 JSON 스키마 필드명(`osc_listen_port`, `target_device_name` 등)과 Rust의 Serde 구조체, Flutter의 모델 클래스가 1:1로 정확히 일치하는가?
    *   UI에서 임의의 로컬 변수에 상태를 저장하지 않고 반드시 Riverpod 전역 상태를 거쳐 갱신되는가?
5. **[Design System] 컴포넌트 나노 체크**:
    *   비활성화된 룸 패널 상단에 정확히 `sigmaX: 5.0, sigmaY: 5.0` 값의 `BackdropFilter`가 적용된 유리막 오버레이가 있는가?
    *   새 룸 생성 시 색상 팔레트 인덱스를 `% 5` 모듈러 연산으로 순환 할당하는 로직이 존재하는가?
6. **[Tasks] 마일스톤 누락 나노 체크**:
    *   현재 작업 중인 Phase에 명시된 체크박스 리스트 중 단 하나라도 누락되거나 미완성된 채로 넘어간 항목이 있는가?
7. **[Convention] 코딩 컨벤션 나노 체크**:
    *   Rust 오디오 코어에 `Vec::new()`, `format!()`, `.clone()` 등 할당 코드가 콜백 루프 안에 단 한 줄이라도 존재하는가? (존재 시 즉각 실패)
    *   파일 I/O 시 단순 문자열 결합(`+`)을 썼는가, 아니면 `std::path::PathBuf`를 정확히 사용했는가?
    *   모든 주석, 로그 출력 텍스트가 100% 한국어로 작성되었는가?
8. **[CI/CD] 객관적 통과 증거 제출**:
    *   테스트, 타입 체크, 린트, 빌드를 모두 통과했음을 증명하는 터미널 콘솔 로그 원문을 제출할 수 있는가?
9. **[Risk] 남은 리스크 파악**:
    *   이 PR을 머지함으로써 발생할 수 있는 잠재적 메모리 누수, 스레드 병목, 예외 상황에 대한 보고서가 작성되었는가?

## 2. 필수 통과 기준 (Mandatory Pass Criteria)

아래 나열된 조건 중 **단 하나라도 충족하지 못하면 실패(Fail)**로 간주하며, 성공할 때까지 스스로 오류를 분석하고 수정을 반복해야 합니다. 타협은 없습니다.

1. **Build (크로스 플랫폼 빌드 통과)**: 
    *   `flutter build macos` 에러 Zero.
    *   `flutter build windows` 에러 Zero.
    *   `cargo build --release` 에러 Zero.
2. **Type Check (타입 안정성 완벽 확보)**:
    *   Flutter(Dart): `dart analyze` 실행 시 `info`, `warning`, `error` 완전 0개. (특히 Null-safety 관련 우회 기법인 `!` 사용 최소화 증명)
    *   Rust: 모든 변수와 반환값이 명시적이고 안전하게(Result/Option) 처리될 것.
3. **Lint (린트 및 포맷팅 무결성)**:
    *   `cargo fmt -- --check` 통과.
    *   `cargo clippy -- -D warnings` 통과 (모든 경고를 에러로 취급하여 통과할 것).
4. **Test (단위/통합 테스트 커버리지)**:
    *   디바운서 로직(250ms 체크) 단위 테스트 통과.
    *   룸 인터록 상태 머신 단위 테스트 통과.
    *   더킹 수식($tanh$, 감쇠율) 단위 테스트 통과.
5. **Migrate (설정 파일 마이그레이션 호환성)**:
    *   기존에 생성된 `config.json` 파일을 새로운 빌드 버전에서 읽었을 때 크래시가 나지 않고 누락된 필드를 디폴트 값으로 안전하게 채워넣을 것.
6. **Security (보안 및 권한)**:
    *   macOS 샌드박스 설정 파일(`DebugProfile.entitlements`, `Release.entitlements`)에 `user-selected.read-only` 권한 태그가 누락 없이 포함되어 파일 피커 에러가 나지 않을 것.
    *   하드코딩된 로컬 테스트 경로가 소스코드에 커밋되지 않을 것.

## 3. 측정 기준 (Measurement Criteria) - 항목별 100점 만점 평가

프로그램의 정상 작동을 보장하기 위한 초정밀 채점표입니다. @Maker 에이전트는 작업 완료 시 아래 리스트의 **각 항목별로 100점 만점 기준**으로 달성 여부를 평가하고 점수를 명시해야 합니다. **단 하나의 항목이라도 100점이 아니라면 통과할 수 없습니다.**

### 🦀 Rust Audio & Network Core
- [ ] **[100점]** 오디오 콜백 내 `String` 또는 `Vec` 힙 메모리 동적 할당 원천 차단 확인.
- [ ] **[100점]** 오디오 콜백 내 디스크 파일 읽기/쓰기(`File::open`) 로직 원천 차단 확인.
- [ ] **[100점]** `std::sync::Mutex` 대기 블로킹 코드가 믹싱 루프에 없음을 확인.
- [ ] **[100점]** OSC 데몬이 0.0.0.0:8000 포트에서 정상적으로 UDP 패킷 비동기 수신.
- [ ] **[100점]** 디바운서(`OscDebouncer`)가 250ms 이내 중복 OSC 주소를 정확히 Drop 처리함.
- [ ] **[100점]** Active 상태가 아닌 룸의 센서 신호가 인입되었을 때 재생 큐에 넣지 않고 즉시 Drop 함.
- [ ] **[100점]** 룸 클리어 신호 수신 시 현재 룸 Cleared -> 다음 룸 Active 상태 머신 전이 확인.
- [ ] **[100점]** SFX 트리거 시 즉시 BGM 상태가 Ducking으로 전환되며 150ms 선형 감쇠 수식 작동 확인.
- [ ] **[100점]** SFX 재생 완료 시 즉시 Unducking으로 전환되며 300ms 선형 복구 수식 작동 확인.
- [ ] **[100점]** 디지털 클리핑 방지를 위한 마지막 버퍼 출력 전 `f(x) = tanh(x)` 수식 적용 확인.
- [ ] **[100점]** `unwrap()`, `expect()` 대신 `Result` 반환 및 에러 매핑 열거형(`AtmosError`) 100% 사용 확인.

### 🎨 Flutter UI/UX & Riverpod
- [ ] **[100점]** 글로벌 헤더 구역 가로 폭 축소 시 "RIGHT OVERFLOWED" 에러(노란 빗금) 발생 안 함. (Wrap/Scroll 래핑 검증)
- [ ] **[100점]** 초당 60회 바뀌는 VU 미터 또는 하단 155px 로그 콘솔 영역이 `RepaintBoundary`로 격리되어 전체 트리 `setState`를 유발하지 않음.
- [ ] **[100점]** 오디오 추가 버튼 클릭 시 macOS 권한 버그(무반응) 없이 네이티브 `file_picker` 창이 정상 오픈됨.
- [ ] **[100점]** 룸 클리어 버튼 내부 글씨(Text) 주변으로 상하 12, 좌우 16 픽셀 패딩 적용 확인.
- [ ] **[100점]** 룸 패널 최상단 이름 부분이 단순 텍스트가 아닌 `TextField` 위젯으로 적용되어 즉시 편집 가능함.
- [ ] **[100점]** 룸 추가 버튼 클릭 시 5가지 테마 컬러 팔레트 번호가 모듈러(`% 5`) 연산으로 겹치지 않게 순환 적용됨.
- [ ] **[100점]** 잠금(Locked) 상태 룸 패널에 `BackdropFilter` 블러(`sigma: 5.0`) 및 반투명 흑색 오버레이가 정확히 렌더링됨.
- [ ] **[100점]** 잠금 오버레이가 덮인 룸 패널 내부의 재생, 볼륨 슬라이더 조작이 완벽히 차단됨(IgnorePointer 적용).
- [ ] **[100점]** 155px 검정색 시스템 로그 콘솔이 새로운 로그 발생 시 자동으로 가장 아래로 스크롤 이동함(`ScrollController.animateTo`).

### 🔗 Architecture & FFI Integration
- [ ] **[100점]** FFI 브릿지(`flutter_rust_bridge`) 자동 생성 코드가 오류 없이 포팅되었음.
- [ ] **[100점]** Riverpod 상태 갱신이 곧바로 FFI 메서 호출을 통해 Rust 메모리로 전달되는 단방향 데이터 흐름 확인.
- [ ] **[100점]** Rust 백엔드에서 생성된 시스템 로그가 StreamSink를 통해 Flutter UI에 즉각 전달되어 터미널에 프린트됨.
- [ ] **[100점]** 파일 경로 결합 시 절대 문자열 덧셈(`+`)을 사용하지 않고, Flutter의 `path_provider`와 Rust의 `PathBuf::join`을 사용하여 경로 슬래시 오류를 방지함.

## 5. 에이전트 자율 운영 규칙 (CI/CD & PR Loop)

`@Maker` 에이전트는 감독관 개입 없이 다음 루프에 따라 기계적으로 움직이며 작업 효율을 극대화해야 합니다.

1. **주기적 감시 체계**: 타이머를 돌려 15분마다 작업 중인 브랜치, PR 상태, 로컬 코드 렌더링 결과를 스스로 검증한다.
2. **CI 에러 스텔스 수정**: 빌드, 타입 체크, 린트(Lint), 테스트 실패 로그를 발견하면 원인을 즉시 분류한다.
   * 단순 린트 오류(들여쓰기, 네이밍 룰), 단순 타입 불일치, 널 세이프티 경고 등은 **디렉터(유저)에게 허락을 구하지 않고 즉각 코드를 수정하여 다시 빌드(Push)한다.**
3. **단순 리뷰 즉각 반영**: 디렉터(유저)가 남긴 UI 텍스트 변경, 색상 코드 변경, 패딩 1~2픽셀 조절 등 단순 리뷰는 묻지 않고 코드에 즉시 반영한다.
4. **⚠️ 에스컬레이션 규정 (디렉터 보고 의무화)**: 코드를 함부로 뜯어고쳐서는 안 되며, **반드시 디렉터(유저)에게 질문하고 승인을 받아야 하는 중대 사항**은 다음과 같다.
   * `config.json`의 DB 구조나 타입을 근본적으로 추가/삭제해야 할 때.
   * macOS 샌드박스, Windows 방화벽 권한 등 OS 보안 정책을 뚫거나 우회해야 할 때.
   * 7대 설계 문서(7 Pillars)와 충돌하는 제품 방향성 변경이 불가피하게 요구될 때.
